# Phone Task Client 方案

日期：2026-06-17

## 背景

当前已有 `phoneworker` 和 `phonecodeworker` 两个终端工具，分别覆盖发码任务和收码任务。随着需求增加，两个工具都逐渐引入了 token、服务器地址、保留设备、轮询间隔、暂停、日志、状态恢复、失败导出等共同能力。如果继续按独立工具扩展，后续新增手机号来源、验证码平台或任务链路时，会重复实现调度、状态、日志和 UI 控制。

本方案新增一个 Windows 客户端程序，从零实现通用任务客户端，不要求复用现有两个工具的代码结构。现有工具可以继续保留，作为线上兼容入口。

## 目标

新增一个统一的 Windows 客户端：`phone-task-client`。

客户端需要支持：

- 功能上平替现有 `phoneworker` 和 `phonecodeworker` 的核心能力。
- 添加和管理多个用户，每个用户填写 OpenAPI token。
- 配置服务器地址、保留设备数量、轮询间隔、请求超时等运行参数。
- 按用户配置 `create_delay`。
- 支持 API 模板，用于配置手机号 API、验证码 API 的请求方式和解析规则。
- 支持任务模板，避免每次创建任务重复填写用户、任务类型、来源和 API 地址。
- 支持发码任务和收码任务。
- 发码和收码都可以使用 API 方式获取手机号，也可以使用 TXT 导入手机号。
- 收码任务可以配置验证码 API。
- UI 展示执行状态、完成情况、错误情况和关键日志。
- 支持暂停、继续、停止。
- 支持程序退出后的状态保存和恢复。
- 支持按手机号快速定位问题。
- 支持导出失败手机号列表和可再次导入的失败文件，方便重新执行。

## 非目标

第一版不做以下能力：

- 不做云端配置同步。
- 不做多机器协同调度。
- 不做自动升级。
- 不做复杂权限系统。
- 不强制替换现有 `phoneworker` 和 `phonecodeworker` 的线上使用方式；新客户端需要先做到功能平替，再灰度替换。
- 不把 bat 中的默认 token 迁移为新客户端默认值。

## 平替能力清单

新客户端第一版需要覆盖现有两个工具的核心功能。

### 平替 `phoneworker`

- 支持 OpenAPI token。
- 支持服务端地址配置。
- 支持取号 API。
- 支持 TXT 导入手机号。
- 支持创建发码任务。
- 支持用户级 `create_delay`。
- 支持全局保留设备数量。
- 支持轮询间隔 `interval`。
- 支持请求超时。
- 支持暂停和继续。
- 支持停止后不再创建新任务。
- 支持日志文件。
- 支持状态展示和错误展示。

### 平替 `phonecodeworker`

- 支持 OpenAPI token。
- 支持服务端地址配置。
- 支持 TXT 导入手机号。
- 支持 API 获取手机号。
- 支持验证码 API。
- 支持验证码 API 文本响应和 JSON 响应解析。
- 支持 GET 查询模板，手机号可通过 `{phone}` 变量传入。
- 支持创建收码任务。
- 支持等待服务端进入验证码阶段。
- 支持拉取验证码并提交。
- 支持提交验证码成功后本地标记成功。
- 支持程序退出后恢复状态。
- 支持失败手机号导出。
- 支持兼容现有格式的可重试失败文件导出。
- 支持按手机号搜索日志和错误。

## 技术选型

使用 Wails + Go 实现。

理由：

- Go 后端适合实现任务调度、HTTP 请求、状态机和本地存储。
- Wails 可以打包为 Windows 桌面程序。
- UI 可以用 Web 技术实现表格、筛选、日志面板和配置页面。
- 后续新增任务来源或验证码来源时，只需要扩展 Go 后端接口和少量 UI 配置项。

## 核心概念

### 任务类型

任务类型只决定服务端创建什么类型的任务：

- `send_code`：发码任务，对应现有发码链路。
- `receive_code`：收码任务，对应现有需要提交验证码的链路。

### 手机号来源

手机号来源和任务类型解耦。发码和收码都可以使用任意手机号来源：

- `api`：从外部 API 获取手机号。
- `txt`：从本地 TXT 文件导入手机号。

### 验证码来源

验证码来源独立于手机号来源：

- `api`：从验证码 API 获取验证码。
- `none`：不需要验证码来源，适用于发码任务。

第一版收码任务使用 `api` 验证码来源。后续可以扩展其他验证码平台。

### 支持的组合

第一版需要支持以下组合：

- 发码 + API 获取手机号。
- 发码 + TXT 导入手机号。
- 收码 + API 获取手机号 + 验证码 API。
- 收码 + TXT 导入手机号 + 验证码 API。

## 配置归属

### 全局配置

全局配置用于控制客户端运行行为和默认调度策略：

- 默认服务器地址。
- 默认保留设备数量。
- 默认轮询间隔 `interval`。
- 默认请求超时 `timeout`。
- 默认日志目录。

`保留设备数量` 是全局配置。创建任务时默认使用全局保留设备数量。

### 用户配置

用户配置用于保存 OpenAPI 身份和该用户的服务端任务参数：

- 用户名称。
- OpenAPI token。
- 可选服务器地址覆盖值。
- `create_delay`。
- 备注。
- 启用状态。

`create_delay` 按用户配置，不放在全局配置里。

### 任务参数取值规则

创建任务时：

- `token` 使用所选用户的 token。
- `base_url` 优先使用用户的服务器地址覆盖值；为空时使用全局默认服务器地址。
- `create_delay` 使用所选用户的 `create_delay`。
- `reserve_devices` 使用全局默认保留设备数量。
- `interval` 使用全局默认轮询间隔。
- `timeout` 使用全局默认请求超时。

### API 模板

API 模板用于保存外部手机号 API、验证码 API 的调用和解析规则，避免每次创建任务重复填写接口细节。

字段：

- 模板名称。
- API 类型：手机号 API / 验证码 API。
- 请求方法：GET / POST。
- URL。
- 请求头，JSON 配置。
- 查询参数，JSON 配置。
- 请求体模板，POST 时使用。
- 手机号或验证码提取规则。
- 未就绪判断规则。
- 成功判断规则。
- 错误信息提取规则。
- 响应类型：JSON / 纯文本 / 自动。
- 备注。
- 启用状态。

手机号 API 第一版默认支持 JSON 返回；纯文本手机号响应作为可配置能力预留。验证码 API 第一版支持 JSON 和纯文本。

模板中可以使用变量：

- `{phone}`：当前手机号，验证码 API 常用。
- `{timestamp}`：当前毫秒时间戳。

变量替换位置：

- URL 路径。
- 查询参数值。
- 请求头值。
- POST 请求体模板。

GET 查询模板是第一版必须支持的能力。例如验证码 API 可以配置为：

```text
method = GET
url = https://example.com/code
query = {"phone":"{phone}"}
```

执行时手机号 `13238381229` 会生成：

```text
https://example.com/code?phone=13238381229
```

也支持把变量直接写在 URL 中：

```text
url = https://example.com/code?phone={phone}
```

如果 URL 自带查询参数，同时模板也配置了查询参数，客户端需要合并两者，并对变量替换后的值做 URL 编码。

### 任务模板

任务模板用于保存日常创建任务的业务表单默认值。

字段：

- 模板名称。
- 默认用户。
- 任务类型：发码 / 收码。
- 手机号来源：API / TXT。
- 手机号 API 模板，选择 API 来源时使用。
- 默认 TXT 文件目录，选择 TXT 来源时使用。
- 验证码 API 模板，选择收码任务时使用。
- 默认失败导出目录。
- 备注。
- 启用状态。

创建任务时优先选择任务模板。使用 API 来源时，选择模板后可以直接开始；使用 TXT 来源时，只需要选择本次 TXT 文件。

### 任务参数快照

任务批次启动后，需要保存用户配置、全局配置、API 模板和任务模板的快照。后续修改配置或模板，不影响已经运行或历史任务。

## UI 设计

### 全局配置页

用于配置客户端运行默认值：

- 默认服务器地址。
- 保留设备数量。
- 轮询间隔。
- 请求超时。
- 日志目录。

配置保存后立即影响后续新建任务。正在运行的任务不动态读取新值，保持任务启动时的参数快照。

### 用户管理页

用于管理 OpenAPI 用户：

- 新增用户。
- 编辑用户名称。
- 编辑 token。
- 设置用户级服务器地址覆盖值。
- 设置用户级 `create_delay`。
- 启用或禁用用户。
- 测试连接。

token 不应明文展示。UI 默认展示脱敏值，需要编辑时重新输入或点击显示。

### API 模板页

用于管理手机号 API 和验证码 API：

- 新增 API 模板。
- 选择 API 类型。
- 配置请求方法、URL、请求头、查询参数和请求体模板。
- 配置响应解析规则。
- 配置未就绪、成功和错误判断规则。
- 测试 API 模板。
- 启用或禁用模板。

测试 API 模板时，UI 需要展示请求摘要、响应状态、响应摘要和解析结果。token、完整验证码等敏感信息默认脱敏。

### 任务模板页

用于管理常用任务配置：

- 新增任务模板。
- 选择默认用户。
- 选择任务类型。
- 选择手机号来源。
- 选择手机号 API 模板或默认 TXT 目录。
- 选择验证码 API 模板。
- 设置默认失败导出目录。
- 启用或禁用模板。

任务模板不保存本次 TXT 文件路径，只保存默认目录，避免下次误用旧文件。

### 创建任务页

字段：

- 任务模板。
- 用户，默认来自任务模板。
- 任务类型：发码 / 收码，默认来自任务模板。
- 手机号来源：API / TXT，默认来自任务模板。

- 不强制取消服务端已经创建的任务。

## 后端模块设计

### `internal/core`

负责通用任务调度和状态推进。

主要职责：

- 管理任务批次。
- 计算可创建任务容量。
- 控制并发补位。
- 处理暂停、继续、停止。
- 驱动任务状态机。
- 定期同步 active 任务。

核心内置 `DevicePoolCoordinator`，负责同一服务端设备池的空闲设备查询和容量分配。

调度逻辑：

```text
循环：
1. 加载当前批次状态。
2. 同步 active 任务状态。
3. 如果暂停，不创建新任务。
4. 由 DevicePoolCoordinator 查询空闲设备。
5. capacity = idle_device_count - global_reserve_devices。
6. 在全局 capacity 允许范围内，为所有运行中的任务批次统一分配创建名额。
7. 对需要验证码的任务获取并提交验证码。
8. 保存状态和事件。
9. 等待 interval 后进入下一轮。
```

### `internal/backend`

负责服务端 OpenAPI 请求。

接口：

- 查询空闲设备。
- 创建发码任务。
- 创建收码任务。
- 查询任务状态。
- 提交验证码。

所有请求需要记录：

- 请求路径。
- 手机号。
- 任务 ID。
- HTTP 状态。
- 耗时。
- 错误。

### `internal/source`

负责手机号和验证码来源。

接口：

```text
PhoneSource.Next(ctx) (phone, ok, error)
CodeSource.Fetch(ctx, phone) (code, error)
```

实现：

- `APIPhoneSource`：从 API 获取手机号。
- `TXTPhoneSource`：从 TXT 导入手机号。
- `APICodeSource`：从 API 获取验证码。

TXT 读取要求：

- 支持 UTF-8 BOM 清理。
- 跳过空行。
- 手机号去重。
- 校验手机号格式。
- 保留原始行号，方便错误提示。

API 响应解析要求：

- 手机号 API 第一版按现有工具经验支持 JSON 响应。
- 验证码 API 支持 JSON 和纯文本。
- 纯文本中提取 4 到 8 位数字验证码。
- 未识别到验证码时标记为未就绪，不立即失败。
- API 模板负责描述字段提取、成功判断、未就绪判断和错误提取规则。

### `internal/store`

使用 SQLite 保存本地状态。

原因：

- 支持程序退出后恢复。
- 支持 UI 表格筛选和手机号搜索。
- 支持历史批次。
- 支持结构化事件日志。
- 比多个 JSON 文件更适合长期维护。

### `internal/security`

负责 token 存储。

第一版建议：

- Windows 下使用 DPAPI 或 Windows Credential Manager 保存 token。
- SQLite 中只保存 token 引用和脱敏展示值。

如果第一版实现成本过高，可以先使用 SQLite 明文存储，但必须在 UI 和文档中标注风险，并预留后续替换接口。推荐直接做 DPAPI，避免后续迁移。

### `internal/logging`

负责日志输出：

- 结构化事件写入 SQLite。
- 文件日志写入本地日志目录。
- UI 订阅实时日志事件。

日志文件按天轮换，每次启动生成新的启动日志文件。程序跨天运行时自动切换到新日期文件。

## 状态机

### 通用状态

`job_items.status`：

- `pending`：待处理。
- `created`：已创建服务端任务。
- `waiting_code`：服务端需要验证码，正在等待验证码。
- `code_submitted`：验证码已提交，等待后续同步。
- `succeeded`：本地认为该手机号处理成功。
- `failed`：失败。
- `stopped`：本地停止后未继续处理。

### 发码任务

第一版发码任务以创建服务端任务成功作为本地成功：

```text
pending
-> create send task
-> succeeded
```

如果后续需要跟踪服务端最终完成状态，可以新增配置项控制。

### 收码任务

收码任务流程：

```text
pending
-> create receive task
-> created
-> remote waiting_promoter_code
-> fetch code
-> submit code
-> succeeded
```

本地语义保持：提交验证码成功后即标记成功，不强制等待服务端最终状态。

如果本地状态是 `code_submitted`，但服务端仍显示需要验证码，并且本地已有验证码，则允许重新提交保存的验证码。

## 错误处理

### 手机号来源错误

- API 获取手机号失败：记录错误，等待下一轮重试。
- TXT 导入格式错误：创建任务前阻断，并提示具体行号。
- TXT 中重复手机号：跳过重复项，记录导入摘要。

### 验证码来源错误

- API 请求失败：记录错误，下一轮继续尝试。
- API 返回无验证码：标记未就绪，下一轮继续尝试。
- API 返回不可解析内容：记录原始响应摘要，下一轮继续尝试。
- 超过服务端任务超时后：按失败处理，并允许重新执行该手机号。

### 服务端错误

- 查询设备失败：不创建新任务，下一轮重试。
- 创建任务失败：记录手机号失败或按错误类型重试。
- 查询任务失败：保留 active 状态，下一轮重试。
- 提交验证码失败：记录错误；如果是超时类错误，可允许重新创建任务重试。

## 并发和补位

所有运行中的任务批次共享同一批服务端设备。客户端必须用全局容量池调度，不能让每个任务批次各自按空闲设备数量独立补位。

同一服务端下，不论有多少账号、多少任务模式同时执行，都必须满足：

- 共用一个全局保留设备配置 `reserve_devices`。
- 共用一次设备空闲查询结果。
- 共用一个全局创建容量 `capacity`。
- 由统一调度器分配创建名额。
- 单个任务批次、单个账号、单个模式都不能直接独立调用设备状态接口后自行补位。

全局容量计算：

```text
capacity = idle_device_count - reserve_devices
```

当 `capacity <= 0` 时，不创建新任务。

设备空闲查询由 `DevicePoolCoordinator` 统一执行。一个调度周期内，同一服务端只查询一次设备状态，查询结果供所有运行中的任务批次使用。这样可以避免多个账号或多个模式同时执行时重复调用服务端 device-status 接口，也避免各自计算 capacity 导致超发。

当同时存在多个运行中的任务批次时，调度器按统一队列分配 `capacity`。第一版建议按任务批次创建时间和任务明细更新时间排序，保证较早批次优先，同时避免某个批次长期占满容量。

当一个任务完成后，不需要等待整批任务完成。下一轮调度发现全局有空闲容量后立即补位。补位频率受 `interval` 控制。

第一版不做无限并发 goroutine。由调度器集中控制创建节奏，避免同时大量请求服务端。

如果未来允许多个不同服务端地址同时运行，则设备池按有效 `base_url` 分组；同一 `base_url` 内仍然共享一个设备池和一个保留设备配置。第一版建议优先使用单一全局服务端地址，避免用户级服务器地址覆盖导致设备池语义混乱。

## 暂停和恢复

暂停状态保存在数据库中，不依赖外部 pause 文件。

暂停后：

- 继续同步已创建任务。
- 继续处理已经进入验证码阶段的任务。
- 不再创建新任务。
- 不再从手机号来源消耗新的手机号。

恢复后：

- 下一轮调度按 `interval` 继续。
- 未处理手机号保持 pending，不会因为暂停被跳过。

## 状态恢复

程序启动后：

- 加载未完成任务批次。
- 从上一次暂停或退出的位置继续执行。
- `pending` 状态继续等待创建。
- `created`、`waiting_code`、`code_submitted` 状态继续同步和推进。
- 对本地已超时且服务端任务可能失效的项，先查询服务端状态，再决定是否重新创建任务。
- 对失败项不自动重跑，除非用户点击重新执行失败项。

如果程序关闭时还有未完成任务：

- 状态保留。
- 下次打开客户端后继续从上一次位置执行。
- 不自动丢弃 pending 手机号。

## 失败导出

每个任务批次支持两类失败导出。

### 失败手机号列表

用于快速查看失败手机号。

格式：

```text
13238381229
18507561351
```

### 可重试失败文件

用于再次导入执行，逻辑需要兼容现有 `phonecodeworker` 的失败文件导出能力。

导出格式根据任务配置生成：

- 如果手机号来源是 TXT，导出失败手机号列表。
- 如果是收码任务，并且需要验证码 API，导出文件第一行写验证码 API，后续每行手机号，兼容现有收码导入格式。
- 如果是发码任务，只导出手机号列表。

导出时清理 BOM 和空白字符。

示例，收码任务失败文件：

```text
https://example.com/code?phone={phone}
13238381229
18507561351
```

如果验证码 API 来自 API 模板，导出第一行应使用任务批次启动时的验证码 API 快照。GET 查询模板需要导出为可再次导入的 URL 形式，优先保留 `{phone}` 变量；如果模板原本是 query 配置形式，则导出时生成等价 URL。

如果验证码 API 模板包含请求头、POST body 或复杂解析规则，单行 URL 无法完整表达。第一版处理方式：

- UI 明确提示“该 API 模板无法导出为兼容 TXT 格式”。
- 仍允许导出失败手机号列表。
- 重新执行失败项时使用任务批次快照，不依赖失败文件表达复杂模板。

## CR 关注点

以下点方案已有默认建议，评审时重点确认是否需要调整：

- 第一版需要功能平替现有 `phoneworker` 和 `phonecodeworker` 的核心能力，再考虑灰度替换。
- 多账号、发码、收码同时运行时，同一服务端下必须共用同一批设备。
- 同一服务端同一调度周期内，设备空闲状态只查询一次，所有任务批次共用查询结果。
- 全局 `reserve_devices` 是唯一保留设备配置，单个账号、任务模板或任务批次不能单独覆盖。
- 创建任务默认通过任务模板快速启动；API 来源可直接开始，TXT 来源只选择本次文件。
- API 模板用于配置手机号 API 和验证码 API 的请求、解析、未就绪和错误规则。
- 失败导出支持失败手机号列表和可重试失败文件；收码任务可导出兼容现有 `phonecodeworker` 格式的失败文件。
- 停止语义是不再创建新的服务端任务，不撤销已创建任务。
- 恢复语义是从上一次暂停或退出位置继续执行。
- 发码任务默认保持“创建成功即成功”的本地语义。
- 收码任务默认保持“提交验证码成功即成功”的本地语义，不等待服务端最终成功状态。
- token 第一版默认使用 Windows DPAPI 或 Windows Credential Manager 保存。
- 失败项重新执行默认创建新的服务端任务，不复用旧任务。
- API 手机号来源第一版默认支持现有 JSON 格式；纯文本手机号响应作为后续扩展。
